## ADDED Requirements

### Requirement: User can create an app
The system SHALL allow users to create a new app with a name and optional description. Each app SHALL have a unique ID generated by the system.

#### Scenario: Create app with name
- **WHEN** user submits a new app with name "My E-Commerce App"
- **THEN** the system creates an app record with a generated UUID, the given name, empty description, and current timestamp

#### Scenario: Create app with name and description
- **WHEN** user submits a new app with name "My E-Commerce App" and description "Testing our checkout flow"
- **THEN** the system creates an app record with both name and description stored

### Requirement: User can list apps
The system SHALL return all apps ordered by creation date descending.

#### Scenario: List apps
- **WHEN** user requests the list of apps
- **THEN** the system returns all apps with id, name, description, and created_at fields

### Requirement: User can update an app
The system SHALL allow users to update an app's name and description.

#### Scenario: Update app name
- **WHEN** user updates app name to "Renamed App"
- **THEN** the system updates the name and returns the updated app record

### Requirement: User can delete an app
The system SHALL allow users to delete an app. Deleting an app SHALL delete all associated versions, scenarios, workflows, and stored Docker image files.

#### Scenario: Delete app with versions
- **WHEN** user deletes an app that has 2 versions with associated scenarios and workflows
- **THEN** the system deletes the app, all its versions, all scenarios scoped to those versions, all workflows scoped to those versions, and all Docker image tar files on disk

### Requirement: User can upload a Docker image as a new version
The system SHALL allow users to upload a Docker image tar file for an app. The system SHALL assign a version tag (user-provided or auto-generated), store the tar file on disk at `/data/app_images/{app_id}/{version_id}.tar`, run `docker load` to import the image, re-tag it to `monkeylab-{app_id}-{version_id}:latest`, and store the image name in the database.

#### Scenario: Upload Docker image tar
- **WHEN** user uploads a 200MB tar file for app "abc-123" with version tag "v1.0"
- **THEN** the system saves the file to `/data/app_images/abc-123/{version_id}.tar`, loads it into Docker, re-tags it to `monkeylab-abc-123-{version_id}:latest`, and creates an `app_versions` record with the image name and version tag "v1.0"

#### Scenario: Upload second version for same app
- **WHEN** user uploads another tar file for the same app with version tag "v1.1"
- **THEN** the system creates a new version record with tag "v1.1", stores a separate tar file, and loads a separately tagged Docker image

### Requirement: User can list versions for an app
The system SHALL return all versions for a given app, ordered by creation date descending, including version tag, Docker image name, and creation timestamp.

#### Scenario: List versions
- **WHEN** user requests versions for app "abc-123" which has 3 versions
- **THEN** the system returns all 3 version records with id, app_id, version_tag, docker_image_name, and created_at

### Requirement: User can delete a version
The system SHALL allow users to delete a specific version. Deleting a version SHALL remove the Docker image tar file from disk, remove the Docker image from the daemon, and delete all scenarios and workflows scoped to that version.

#### Scenario: Delete version
- **WHEN** user deletes version "v1.0" of an app
- **THEN** the system removes the tar file, runs `docker rmi` on the image, deletes associated scenarios and workflows, and removes the version record

### Requirement: App management UI page
The system SHALL provide an Apps page accessible from the sidebar navigation. The page SHALL list all apps, allow creating new apps, and allow navigating to an app's detail view showing its versions.

#### Scenario: Navigate to Apps page
- **WHEN** user clicks "Apps" in the sidebar
- **THEN** the system displays a list of all apps with create, edit, and delete actions

#### Scenario: View app detail
- **WHEN** user clicks on an app in the list
- **THEN** the system shows the app's versions with upload, delete actions, and links to scenarios/workflows for each version
